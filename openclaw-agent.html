OpenClaw Agent

深度解析生产级 Coding Agent 的工程架构与实现原理

核心认知： 主流框架（LangChain/Dify）在 Demo 阶段表现优异，但在生产环境中面临行为不可控、排错困难等挑战。
graph TD A[EventStream] --> B[Agent Loop] B --> C[Context Engine] C --> D[Parallel Tools] D --> E[MEMORY.md] E --> B

为何自研 Agent？

生产环境下的痛点直接驱动了架构的演进：

框架方案 (Demo 级)
  • 链式调用: 难以处理复杂往复。
  • 过度抽象: 细节调整需翻阅多层库。
  • 黑盒调试: 堆栈极深，排错成本高。
生产级方案 (OpenClaw)
  • EventStream: 生命周期事件流，解耦 UI。
  • Context Engine: 智能组装与 Token 管理。
  • 文件级记忆: MEMORY.md 易读可编辑。

五阶段执行模型

OpenClaw 在 pi-momo 的基础上引入了更严谨的执行流程：

Agent Loop 深度解析

理解 Agent 与 ChatBot 的本质区别：一行代码解决循环逻辑。

// 简化自 pi-mono agent-loop.ts
async function agentLoop(messages) {
  while (true) {
    const response = await llm.chat(messages);
    yield { type: 'message_end', response };

    // 核心差异：检测到 tool_calls 则继续循环
    if (!response.toolCalls?.length) break;

    const results = await Promise.all(
      response.toolCalls.map(call => executeTool(call))
    );
    yield { type: 'tool_execution_end', results };
    messages.push(...results);
  }
}
并行执行是关键：
使用 Promise.all 将多个独立工具调用的耗时从 O(N) 降低到 O(1)。

工业级 RAG 管线

按语义边界分块

利用 AST (Tree-sitter) 识别函数和类边界，确保代码片段的完整性。

混合检索 + RRF 融合

结合向量检索（语义）与 BM25（关键词），使用 RRF 算法进行排名融合。

Cross-Encoder 精排

召回 Top-20 后，使用精排模型将 Top-5 注入上下文，大幅减少噪声。

记忆架构：Dreaming 系统

受睡眠启发的记忆巩固机制，自动整理 MEMORY.md。

graph LR A[当日对话] --> B(Light Sleep: 提取) B --> C(Deep Sleep: 合并冲突) C --> D(REM: 跨主题关联) D --> E[DREAMS.md / MEMORY.md]

Compaction 策略： 当 Token 接近预算时，执行摘要压缩，但必须保留文件路径、函数名等标识符。

GBrain 高级记忆系统

针对大规模记忆场景，采用 Compiled Truth + Timeline 模式：

  • Compiled Truth: 动态更新的当前理解。
  • Timeline: 支撑理解的不可变证据链。

这种模式避免了向量检索中新旧信息冲突导致的认知混乱。

多 Agent 与子进程隔离

通过派生 SubAgent 解决上下文污染问题：

const result = await spawnSubagent({
  task: '分析 auth.py 模块',
  tools: ['Read', 'Grep'],
  budget: { maxTokens: 16000 }
});
// 主 Agent 只接收结果，不被中间分析过程干扰
